## Problem Statement

MoodTune v3 is a Streamlit app. It works. But Streamlit is a ceiling: no real chat UX,
no per-user persistence, no streaming, no polish. The v4 design from yesterday locked
in Python FastAPI + React + Chroma + Anthropic. Today the scope changed twice:

1. **Free tier only.** No Anthropic API, no paid vector DB, no paid hosting.
2. **Codex writes the code, Claude reviews later.** You have Codex Business tokens
   to burn but limited Claude Pro budget. Claude becomes a post-ship quality pass,
   not the builder.

The deliverable is a **new repo, clean git, fully TypeScript, Codex-built** — plus
the phase-by-phase `codex exec` prompts, the Claude Code skills/agents to invoke for
the later polish pass, and the workflow that makes the two agents cooperate without
stepping on each other.

## What Makes This Version Different

1. **Zero Python at runtime.** One language (TS), one deploy (Vercel), one DB (Supabase).
2. **Runs entirely inside free tiers.** Gemini 2.5 Flash-Lite (LLM), local
   Transformers.js (embeddings — no API), Supabase pgvector (500MB), Vercel Hobby.
3. **Codex-first authoring.** Six phase prompts build the whole app. You paste,
   review, commit. No interactive pairing.
4. **Claude as auditor, not author.** Separate post-ship skill plan (Section 10)
   runs AFTER Codex ships v1.
5. **Taste profile + journey + chat + feedback loop** — full v4 feature set from
   yesterday's design, ported to TS.

## Constraints

- Free tier only. No card on file for any provider.
- Codex (OpenAI) writes the code via `codex exec`. Claude Code (Anthropic) plans and
  reviews only — never runs `codex exec` itself.
- New GitHub repo (`Krishpotanwar/MoodTune-v4`). Streamlit v3 stays frozen.
- Dataset: existing Kaggle 114k CSV, pre-filtered to ~15k rows (~6MB) before committing
  at `data/spotify_tracks.csv`. **Do NOT use Git LFS** — Vercel Hobby's LFS support is
  historically flaky and GitHub's 1GB/month LFS bandwidth is consumed by every Vercel
  build pull. The filter step is a 15-line pre-seed script (`scripts/filter-csv.ts`)
  that takes the raw 114k Kaggle CSV, keeps top-5k popularity + 5k stratified-by-genre
  + 5k uniform-random, and writes out ~6MB. Commit the filtered file plain. Raw 114k
  stays on local disk only.
- Target timeline: **18–25 days at 4–6 hrs/day** (revised from 10–14 — learning curve
  on `@huggingface/transformers`, `@ai-sdk/google`, pgvector, Upstash, and Next 15
  App Router is non-trivial for a first-time user). 10 days is demo-ready if everything
  goes smoothly; budget the buffer anyway.
- College demo deadline is the real deadline. Portfolio polish is bonus.

## Premises (all agreed, post-review corrections folded in)

1. **Next.js 15 App Router + TypeScript + Tailwind + shadcn/ui + Vercel AI SDK (pinned).**
   Pin `ai@^4.0.0` and `@ai-sdk/google@^1.0.0` to avoid breaking-change churn.
2. **LLM = Google Gemini 2.5 Flash-Lite** via `@ai-sdk/google` (free tier, 15 RPM /
   1000 RPD / 250k TPM). Verify exact model string at `aistudio.google.com` before
   Phase 1 — may be `gemini-2.5-flash-lite` or include a preview suffix.
3. **Embeddings = local Transformers.js (primary), quantized model bundled with deploy.**
   Use `@huggingface/transformers` (new package name; `@xenova/transformers` is
   deprecated). Model `Xenova/all-MiniLM-L6-v2`, 384-dim, **quantized (q8) variant
   only** (`onnx/model_quantized.onnx`, 22.9 MB — verified via HF API). The fp32
   variant is 90.4 MB and flirts with GitHub's 100MB-per-file hard cap; quantized
   halves cold-start, barely affects semantic-similarity quality, and is safe to
   commit plain. Commit to `public/models/Xenova/all-MiniLM-L6-v2/` (only the
   quantized ONNX + tokenizer.json + config.json + tokenizer_config.json + special
   tokens — NOT the 90MB fp32 file). Load with
   `env.localModelPath = './public/models'` + `env.allowRemoteModels = false` +
   `pipeline('feature-extraction', 'Xenova/all-MiniLM-L6-v2', { dtype: 'q8' })`.
   Rationale: Vercel serverless does NOT persist `./.cache/` between cold starts —
   unbundled, every cold lambda re-downloads 25MB and blows the latency budget.
   Bundling keeps the whole MiniLM pipeline inside the function package (compressed
   ~10MB, well under Vercel Hobby's 50MB-compressed function size cap after
   tree-shaking onnxruntime-node). Runs identically in Node on Vercel and locally
   for the seed script. ZERO API calls. HuggingFace Inference API was deprecated
   for sentence-transformers feature-extraction in 2024 — do not rely on it.
   **Pre-Phase-1 proof-of-feasibility: the Assignment smoke test now requires a real
   cold-start measurement on Vercel (see section at the bottom), not just a local bench.**
4. **Database = Supabase free tier**, single Postgres with **pgvector** extension.
   Stores vectors + sessions + feedback + taste profiles + explanation cache + chat.
   500MB cap. **Chat + cache tables are pruned daily by the keepalive cron** (7-day
   retention for `chat_messages`, 30-day for `explanation_cache`) to avoid hitting
   the ceiling. Migrations run via **Supabase CLI** (`supabase db push`), NOT via the
   JS client. Requires `SUPABASE_DB_URL` or `supabase link` with project ref. **RLS
   is enabled on every user-scoped table in the initial migration** — no public-demo
   "Phase 6" defer. The anon JS client can only see rows belonging to its own
   session cookie; the service-role client (server-only) bypasses RLS for admin ops.
5. **Seed corpus ≈ 12–13k stratified tracks.** Filter script targets 15k via three
   samplers (5k top-popularity + 5k stratified-by-genre + 5k uniform random), then
   dedups by (lowercase(name), lowercase(artist)). Expect ~12-13k rows after dedup —
   popular songs leak into the genre-stratified bucket. That's fine: still 2.5x the
   original "10k popular" plan, with actual long-tail coverage. Pure top-popularity
   kills queries like "funeral doom drone" or "shoegaze rainy". DB footprint ≈ 50MB
   for 13k rows × 384-dim q8-embeddings + metadata. **Seed time budget: 20–30 minutes**
   on a modern laptop (~80ms/row ONNX inference, batched 500-row Supabase inserts).
   Slower laptops: 45 min. Quantized (q8) model also halves inference time vs fp32.
6. **Codex (OpenAI) writes 100% of runtime code.** Claude Code runs LATER for review,
   refactor, security, test coverage, a11y. Separate skill plan in Section 10.
7. **Vercel Cron pings Supabase DAILY.** Vercel Hobby crons require daily-minimum
   cadence (changed Oct 2024). Daily keepalive easily prevents the 7-day inactivity
   pause.
8. **Distributed rate limiting via Upstash Redis free tier** (currently 500k
   commands/month and 256MB data, free, no card). In-memory Maps are useless across
   serverless instances; Upstash fixes it and is the one extra service worth adding.
9. **Spotify Client Credentials** for live enrichment (album art, preview URLs) — free
   app keys only, no user login.
10. **Codex CLI must be installed and reachable on PATH before Phase 1.** Verified on
    2026-04-17 that this machine has `~/.codex/config.toml` (left from a prior install)
    but NO `codex` binary on PATH. Install step (do this weekend, before The Assignment):
    `brew install openai/tap/codex` or `npm install -g @openai/codex-cli` — whichever

## Approaches Considered

### Approach A: 6 Phased Codex Prompts (RECOMMENDED)
One `codex exec` per phase. Each phase produces a working increment. You commit after
each. Phase 5 is split (5a = journey+viz, 5b = Spotify+cron+deploy) because v1 had
Phase 5 at ~200 lines of prompt — too big for a single exec.
- **Effort:** S (30–60 min/phase for Codex, 15 min/phase for your review)
- **Risk:** Low — bounded failure per phase
- **Pros:** fits Codex context, checkpoint per phase, easy rollback
- **Cons:** 6 paste actions instead of 1

### Approach B: Codex-Managed TASK.md
One planning prompt → Codex writes TASK.md → `codex exec "do next item"` loop.
- **Effort:** S upfront, burns tokens
- **Risk:** Medium — drift over long chains
- **Cons:** hard to debug when mid-chain step regresses

### Approach C: Feature Branch + PR per Phase
Branch + PR per phase.
- **Effort:** M
- **Cons:** overhead without team benefit

**CHOSEN:** Approach A with 6 phases.

## Data & Schema

### Postgres (Supabase) — `0001_init.sql`

```sql
create extension if not exists vector;

-- Track catalog: public-readable, no write from anon.
create table tracks (
  id            text primary key,
  name          text not null,
  artist        text not null,
  genre         text,
  popularity    int,
  valence       real,
  energy        real,
  tempo         real,
  acousticness  real,
  danceability  real,
  spotify_uri   text,
  album_art     text,
  preview_url   text,
  embedding     vector(384),
  embedding_pending boolean default false,
  created_at    timestamptz default now()
);
create index tracks_embedding_idx
  on tracks using hnsw (embedding vector_cosine_ops);
create index tracks_embedding_pending_idx
  on tracks (embedding_pending) where embedding_pending = true;

create table sessions (
  id          uuid primary key default gen_random_uuid(),
  created_at  timestamptz default now()
);

create table taste_profiles (
  session_id     uuid primary key references sessions(id) on delete cascade,
  summary        text not null,
  rating_count   int  not null default 0,
  updated_at     timestamptz default now()
);

create table feedback (
  id          bigserial primary key,
  session_id  uuid not null references sessions(id) on delete cascade,
  track_id    text not null references tracks(id),
  rating      int  not null check (rating in (-1, 1)),
  created_at  timestamptz default now()
);
create index feedback_session_idx on feedback (session_id, created_at desc);

create table explanation_cache (
  cache_key   text primary key,
  explanation text not null,
  created_at  timestamptz default now()
);
create index explanation_cache_created_idx on explanation_cache (created_at);

create table chat_messages (
  id          bigserial primary key,
  session_id  uuid not null references sessions(id) on delete cascade,
  role        text not null check (role in ('user','assistant')),
  content     text not null,
  created_at  timestamptz default now()
);
create index chat_messages_session_idx on chat_messages (session_id, created_at);
create index chat_messages_created_idx on chat_messages (created_at);

-- RLS: anon JS client is locked down. All user-scoped reads/writes go through
-- server routes with the service-role key (which bypasses RLS). Anon can read
-- tracks only.
alter table tracks             enable row level security;
alter table sessions           enable row level security;
alter table taste_profiles     enable row level security;
alter table feedback           enable row level security;
alter table explanation_cache  enable row level security;
alter table chat_messages      enable row level security;

create policy tracks_anon_read  on tracks           for select to anon using (true);
-- No anon policies on the other tables: default-deny. The service-role key
-- (used by server-side supabaseAdmin) bypasses RLS entirely.
```

**Why no per-row `request.headers->mt_sid` policy?** Simpler and safer: anon never
touches user-scoped tables at all. Every user-scoped operation goes through a Next.js
API route that reads the `mt_sid` cookie server-side and uses the service-role
client. No chance of a client bug exposing cross-session data via direct Supabase
JS calls. The `NEXT_PUBLIC_SUPABASE_ANON_KEY` exists only so the catalog can be read
without a server hop (optional optimization; not used in v1 but kept open).

### `0002_search_rpc.sql`

```sql
create or replace function match_tracks(
  query_embedding vector(384),
  match_count int default 20
)
returns table (
  id text, name text, artist text, similarity float,
  spotify_uri text, album_art text, preview_url text,
  valence real, energy real
)
language sql stable as $$
  select id, name, artist,
         1 - (embedding <=> query_embedding) as similarity,
         spotify_uri, album_art, preview_url, valence, energy
  from tracks
  where embedding is not null
  order by embedding <=> query_embedding
  limit match_count;
$$;
```

### Journey algorithm

```typescript
// lib/journey.ts  — pseudocode, implement in TS

import { embed } from './embeddings';
import { supabaseAdmin } from './supabase';

export async function generateEmbeddingJourney(
  startMood: string,
  targetMood: string,
  steps: number = 15,
  tasteProfile?: string
): Promise<JourneyResult> {
  const startEmb = await embed(startMood);
  const targetEmb = await embed(targetMood);
  const tasteEmb = tasteProfile ? await embed(tasteProfile) : null;
  const seen = new Set<string>();
  const tracks: Track[] = [];
  const pathPoints: [number, number][] = [];

  for (let i = 0; i < steps; i++) {
    const t = i / (steps - 1);                // 0.0 → 1.0

    // Linear interpolation start → target
    let waypoint = lerp(startEmb, targetEmb, t);

    // Taste blending: triangle weight peaks at midpoint
    if (tasteEmb) {
      const blend = 0.3 * (1 - Math.abs(2 * t - 1));   // 0 → 0.3 → 0
      waypoint = lerp(waypoint, tasteEmb, blend);
    }

    waypoint = l2normalize(waypoint);

    // Query pgvector, skip already-seen IDs
    const { data } = await supabaseAdmin.rpc('match_tracks', {
      query_embedding: waypoint, match_count: 5
    });
    const pick = data.find((t: Track) => !seen.has(t.id));
    if (!pick) continue;

    seen.add(pick.id);
    tracks.push(pick);
    pathPoints.push([pick.valence ?? 0.5, pick.energy ?? 0.5]);
  }

  return { tracks, path_points: pathPoints };
}

function lerp(a: number[], b: number[], t: number): number[] {
  return a.map((v, i) => v * (1 - t) + b[i] * t);
}

function l2normalize(v: number[]): number[] {
  const norm = Math.sqrt(v.reduce((s, x) => s + x * x, 0)) || 1;
  return v.map(x => x / norm);
}
```

`path_points` is simply `[valence, energy]` per track — no PCA, no extra libs.
Used by `<JourneyViz>` as Recharts `(x, y)` coordinates.

## Section 8.5: Mood-Text Builder (Embedding Input)

**This is the most important function in the app.** The embedder (MiniLM) sees only
text. If we feed it `"Shake It Off by Taylor Swift. Genre: pop."` the embedder has no
idea the song is high-energy, fast, danceable. The query `"high energy euphoric"` will
match by genre association, not actual mood. To fix this we translate the Spotify
audio features (valence/energy/tempo/danceability/acousticness) into natural-language
descriptors BEFORE embedding.

```typescript
// lib/mood-text.ts

export function buildMoodText(track: {
  name: string;
  artist: string;
  genre?: string | null;
  valence?: number | null;       // 0..1 — musical positivity
  energy?: number | null;        // 0..1 — intensity + activity
  tempo?: number | null;         // BPM
  danceability?: number | null;  // 0..1
  acousticness?: number | null;  // 0..1
}): string {
  const parts: string[] = [`${track.name} by ${track.artist}.`];
  if (track.genre) parts.push(`${track.genre}.`);

  const mood = describeValence(track.valence);
  const intensity = describeEnergy(track.energy);
  const pace = describeTempo(track.tempo);
  const dance = describeDance(track.danceability);
  const texture = describeAcoustic(track.acousticness);

  const descriptors = [mood, intensity, pace, dance, texture].filter(Boolean);
  if (descriptors.length) parts.push(descriptors.join(", ") + ".");
  return parts.join(" ");
}

function describeValence(v?: number | null): string {
  if (v == null) return "";
  if (v < 0.2) return "melancholic, sad, somber";
  if (v < 0.4) return "introspective, subdued";
  if (v < 0.6) return "bittersweet, reflective";
  if (v < 0.8) return "upbeat, cheerful";
  return "euphoric, joyful, bright";
}

function describeEnergy(e?: number | null): string {
  if (e == null) return "";
  if (e < 0.2) return "calm, still, gentle";
  if (e < 0.4) return "relaxed, soft";
  if (e < 0.6) return "moderate intensity";
  if (e < 0.8) return "energetic, driving";
  return "high energy, intense, aggressive";
}

function describeTempo(t?: number | null): string {
  if (t == null) return "";
  if (t < 70)  return "slow tempo";
  if (t < 100) return "medium-slow tempo";
  if (t < 130) return "medium tempo";
  if (t < 160) return "fast tempo";
  return "very fast tempo";
}

function describeDance(d?: number | null): string {
  if (d == null) return "";
  if (d < 0.3) return "not danceable";
  if (d < 0.7) return "";
  return "danceable, groovy";
}

function describeAcoustic(a?: number | null): string {
  if (a == null) return "";
  if (a > 0.7) return "acoustic, organic";
```

   directly; only track-side embeddings go through `buildMoodText`.
